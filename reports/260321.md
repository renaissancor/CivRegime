Key patterns captured:
- **Ruling class vs population majority**: Qing (~2% Jurchen over 95%+ Han), Jin Dynasty (Jurchen ruling class), Ostrogoths (~10% over Latin majority)
- **Ethnic survival under conquest**: Persian language survived 6 conquests (Arab → Turkic → Mongol); Berbers Islamized but retained identity; Coptic Christians in Egypt
- **Religion as identity splitter**: South Slavic → Serb/Croat/Bosniak split via Ottoman/Byzantine/Habsburg patronage; Iberia: Muslim/Christian/Jewish communities on same territory
- **Nomadic overlay pattern**: Steppe peoples ruled sedentary majorities without displacing them (Mongol, Jurchen, Hunnic, Turkic cycles)
- **Territory as crossroads**: Transoxiana changed hands 10+ times (Achaemenid → Greek → Parthian → Sassanid → Arab → Persian → Turkic → Mongol → Timurid → Uzbek)

### Open Question: RDBMS vs JSON flat files
Conclusion: JSON is correct. The data is a property graph — typed, annotated, irregular edges. RDBMS would require 10+ junction tables before any data. JSON fits in memory, schema evolves freely. Revisit only if concurrent writes or millions of records needed.

## Session 2 (this session)

### Architecture — scalable directory structure
Split flat `regimes.json` / `successions.json` into per-region directories.
Taxonomy trees (religions, languages, ethnicities) moved to filesystem hierarchies — parent derived from path, not stored in file.

### Territory model redesign — two-level split
Separated `territory` (broad political zone for succession analysis) from `region` (atomic map polygon with simple control timeline).

- **territory**: keeps rich metadata + ethnic composition periods
- **region**: new entity, GeoJSON Feature format (`.geojson` files), contains only `{territory FK, periods[{start, end, regime}], geometry: null}`

Reasoning: map renderer needs fine-grained stable polygons (duchy/province level). Succession analysis needs broader cultural zones. These are different granularities. Separating them keeps both clean.

GeoJSON chosen over plain JSON for regions because:
- `d3.geoPath()` reads it natively
- `geometry: null` is valid GeoJSON — placeholder until polygons are added
- GitHub renders `.geojson` files as interactive maps
- GIS tools (QGIS etc.) can open directly

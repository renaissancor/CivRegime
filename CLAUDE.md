# CivRegime — Project Primer

CivRegime is a CSV-first historical-political dataset. **Schema is two-tier:**

- **Civilization** (top tier; was `state` until the 2026-05-01 rename — still mid-migration in the working tree)
- **Polity** (entity tier; a political entity at the intersection of ethnicity × territory × ideology × time)

**Dynasty** (~192 records) is a **cross-cutting tag** via the `polity_dynasty` junction, *not* a third tier. The earlier "Regime" tier was **dropped** on 2026-05-01 — identity drift inside long-running entities (Roman Republic → Pagan Empire → Christian Empire → Byzantine) is handled by polity-splits at major transitions, not sub-polity periods.

Auxiliary entities: `ethnicity`, `language`, `religion`, `territory`, `figure`, `government`, plus planned `city` (see `docs/model/city.md`).

## CSV-first architecture — sources of truth

**`csvs/` and a small set of taxonomy filesystem trees are the only sources of truth.** Everything else in `data/` is generated and must not be hand-edited.

| Source of truth | What it holds |
|---|---|
| `csvs/*.csv` | Entities (civilization, polity, dynasty, figure, government, …) and relations (`polity_dynasty`, `polity_succession`, `polity_territory`, …) |
| `data/languages/` (FS tree) | Language phylogeny — parent derived from path |
| `data/religions/` (FS tree) | Religion taxonomy — parent derived from path |
| `data/territories/{region}/` (FS tree) | Territory grouping by region |

**Generated outputs (do not hand-edit):**

- `data/*.json` per-entity files — generated by `code/makejson/*.js` from the sources above
- `civregime.db` (DuckDB) — loaded via `scripts/load_csvs.sql` for analytical queries

**The four rules that fall out:**

1. **Edit only sources, never generated JSON.** Changes to entities go in `csvs/*.csv` (or the relevant taxonomy file); regeneration produces the JSON. If you find yourself editing `data/<entity>.json`, stop and find the CSV.
2. **Every CSV must have a logical description** in `docs/model/<concept>.md` — purpose, columns, semantics, FK relationships. A CSV without a model doc is an incomplete commit.
3. **Every CSV must have a generator.** A CSV that no `code/makejson/*.js` consumes is dead data.
4. **Workflow:** edit a CSV row → run regenerate-all → project state updates. Reviews are CSV / taxonomy diffs; JSON diffs are generated noise.

The taxonomy FS trees (`data/languages/`, `data/religions/`, `data/territories/`) are a **deliberate exception** because hierarchy is expressed by path. Do not propose flattening them into CSVs unless the user explicitly asks.

## Conventions

- **IDs:** snake_case, stable, never reused
- **Years:** integer; BCE is negative (`-753` = 753 BCE); empty `end` field = ongoing
- **Free-text in CSVs:** avoid commas inside fields — use semicolons or em-dashes
- **Foreign keys:** plain id strings (e.g., `polity.civilization_id` → `civilization.id`)
- **Word "regime"** is acceptable only in: brand mentions (`civregime`, `CivRegime`), English-noun prose ("Pol Pot regime"), and historical reports/snapshots. **Never as an entity term in new code or schema.**

## Schema discipline — FK-driven scoping

Entity tables earn rows by inbound foreign keys, not historical fame. A row belongs in `city`, `figure`, etc. only when something else in the dataset points to it. No decorative reference rows. When seeding, walk the *references* first and back-fill entities. Canonical statement: `docs/model/city.md` *Scope Discipline*.

## Active migration

`state` → `civilization` rename is **largely done in the working tree** but uncommitted (~198 modified files as of 2026-05-02). If you see `state_id` / `state.csv` / `data/states.json` references that conflict with `civilization_*` equivalents, treat them as in-flight artifacts to be cleaned up, not design conflicts.

## What this project is *not*

CivRegime is an **architecture for curation**, not a complete dataset. Coverage is uneven; sparse panels are expected. Decisions are driven by political-history queries, not by exhaustive cataloging.
